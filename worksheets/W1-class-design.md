## 2. Class Definition

ระบุ class ที่จะใช้ + เกณฑ์ที่ทำให้แต่ละ class แตกต่าง:

| # | ชื่อ Class | คำอธิบาย (3-4 ประโยค) | สิ่งที่ทำให้ต่างจาก class อื่น |
|---|---|---|---|
| 1 | เป๋า | กำมือแน่น เป็นลักษณะกลมลักษณะมือแน่น นิ่ง ไม่มีนิ้วเหยียด|มือนิ่งมากไม่ขยับsensorแทบไม่เห็นการเคลื่อนที่เลย |
| 2 |ยิ้ง |เหยียดนิ้วชี้และนิ้วกลางออก นิ้วที่เหลือกำ  |แสดงท่าsensorเห็น spike สั้นๆ ก่อนนิ่ง |
| 3 | ชุ่บ|แบมือเปิดออก นิ้วทุกนิ้วเหยียดตรง ฝ่ามือหงายหรือคว่ำขนานพื้น |มือแบออกทำให้มุมของมือต่างจากเป๋าและยิ้งอย่างชัดเจนsensorจ |

## 3. Edge Cases — สิ่งที่อาจสับสน

ระบุสถานการณ์ที่อาจทำให้โมเดลสับสน:

```
1. ยิ้งยังไม่ถึงท่าอาจถูกอ่านเป็นเป๋า
   เพราะช่วงที่นิ้วยังไม่เหยียดเต็มที่ sensor จะเห็นคล้ายกำมือ

2. ฉุบตั้งตรงอาจถูกอ่านเป็นเป๋า
   ถ้าแบมือแต่ตั้งข้อมือตรง แกน Z จะสูงคล้ายเป๋า

3. ขยับมือขณะเล่นอาจเป็น noise
   ถ้าเก็บ sample ตอนเคลื่อนไหว ไม่ใช่ตอนค้างท่า
```

**วางแผนแก้:**
```
[เขียนวิธีจัดการ edge cases]
กำหนดให้ระบบอ่านค่าเฉพาะช่วง 0.5 วินาทีหลังจากกดปุ่ม trigger — บังคับให้ผู้เล่นค้างท่าก่อน ไม่ใช่อ่านตลอดเวลา
```

## 5. Diversity Plan — กันอคติ (Bias)

ตอบ 3 คำถาม:

### a) มีหลาย "environment" ไหม?
```
[เก็บในห้อง ทั้งนั่งและยืนมือสูงต่างระดับอกและโต๊ะ
```

### b) มีหลาย "subject" ไหม? (กรณี Vision/Audio/Motion)
```
[สมาชิกทั้ง 3 คนในทีมเก็บคนละ 40 samples/class — ได้ความหลากหลายของขนาดมือ
```

### c) มีหลาย "variation" ไหม?
```
[เก็บทั้งท่าช้าๆและเร็วแบบเล่นจริงเพื่อให้โมเดลรับมือได้ทั้งสองแบบ
```

## 7. Success Criteria

ทีมจะถือว่าโมเดลใช้งานได้เมื่อ:

```
[ ] Accuracy V1 ≥ 75%
[ ] V2 ดีกว่า V1 อย่างน้อย 1%
[ ] False negative ของ class สำคัญ ≤ 20%
[ ] Inference ภายใน 500 ms
```
